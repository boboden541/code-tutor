# Contract-first: генерация API-контракта ВПЕРЁД из требований

> Шов между фронтом и бэком в fullstack-книге (и контракт для внешних потребителей в backend-проекте). Контракт фиксируется ДО кода и становится источником правды: бэк реализует ровно его, фронт строит типы ровно под него.
>
> **Отличие от навыка `openapi-spec`.** Тот строит OpenAPI *обратно* — из существующего кода (якоря routing/contract/model). Здесь кода ещё нет: контракт строится *вперёд* из сущностей и эндпоинтов Файла №2. **Поле-модель и валидацию заимствуем** из `openapi-spec` (`resources/openapi_field_model.md`, `resources/openapi_validation_checklist.md`), а источник данных — требования, а не код.

## Что заимствовать из openapi-spec (без изменений)

- **Поле-модель** «свойство → ключ OpenAPI» (`type`/`format`, `required`, `nullable`, `pattern`, `min/maxLength`, `min/maximum`, `enum`, `example`).
- **Версия 3.0.3 по умолчанию** и кодирование `nullable: true` рядом с одиночным `type` (не union-тип 3.1).
- **Структурность:** один DTO = одна схема в `components/schemas`, переиспользование через `$ref`.
- **Объективная валидация** внешним линтером с циклом самокоррекции (см. ниже).
- **Один файл на домен:** `book/contracts/<domain>.yaml`, методы дописываются накопительно.

## Что иначе (forward-режим)

| Аспект | openapi-spec (reverse) | contract_first (forward) |
|--------|------------------------|---------------------------|
| Источник полей | DTO/Entity в коде | сущности и поля из Файла №2 (раздел FR-2) |
| Источник эндпоинтов | роутинг в коде | список эндпоинтов из Файла №2 (раздел IR-1) |
| Ограничения полей | валидаторы в коде | бизнес-правила из Файла №2 (раздел FR-4) |
| Коды ответов | обработка ошибок в коде | краевые случаи из Файла №2 (раздел UR-3) |
| `[NEEDS_INVESTIGATION]` | поле не прослежено в коде | требование неоднозначно/не задано → `[NEEDS_DECISION]` |

## Процедура (для `/tutor-stack`, Этап 5)

1. **Сущности → схемы.** Из FR-2 Файла №2 для каждой сущности собери `components/schemas/<Entity>` (поля, типы, обязательность, nullable, ограничения из FR-4). Раздели на схемы запроса/ответа, если они отличаются (`<Entity>Create`, `<Entity>Read`).
2. **Эндпоинты → operations.** Из IR-1 для каждого пути+метода добавь операцию в `paths`: параметры, `requestBody` (если есть тело), ответы.
3. **Коды ответов.** Заложи из UR-3/IR-2: `200/201/204`, и ошибки `400/401/403/404/409/422` там, где их предполагают краевые случаи.
4. **Аутентификация.** Если в требованиях есть auth — добавь `securitySchemes` (например `bearerAuth`) и применяй к защищённым операциям.
5. **Неоднозначности** помечай `[NEEDS_DECISION]` в `description`, не выдумывай ограничения.
6. **Валидация (обязательно).** Прогон линтера через Bash, цикл «правка → перепроверка», предел 5 итераций:
   - основной: `npx --yes @redocly/cli@latest lint --extends minimal book/contracts/<domain>.yaml`
   - fallback: `python -m openapi_spec_validator book/contracts/<domain>.yaml`
   - деградация: `python -c "import yaml,sys; yaml.safe_load(open(sys.argv[1]))" book/contracts/<domain>.yaml` + предупреждение.

## Как контракт используется дальше

- **План (`/tutor-plan`):** каждый эндпоинт IR → служебный шаг `define_api_contract` (ссылка на контракт) + шаг бэк-реализации + шаг фронт-потребления.
- **Запись (`/tutor-write`):**
  - бэк-шаги: реализуют путь/схему **точно по контракту** (роутер отдаёт ровно описанные коды и форму).
  - фронт-шаги: `api_client` строит типы DTO из схем контракта; компоненты потребляют их.
- **Валидация (`/tutor-validate`):** проверка `contract-drift` — формы/коды в коде шагов соответствуют контракту; типы фронта соответствуют схемам.

> Контракт — «договор», а не документация постфактум. Меняется требование → сначала правится контракт (повторный `/tutor-stack`), потом код обеих частей.
