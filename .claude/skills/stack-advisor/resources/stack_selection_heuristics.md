# Эвристики выбора и валидации стека

> Для `/tutor-stack`, Этап 2. Помогают (а) предложить стек, когда он не задан, и (б) валидировать стек, заданный пользователем. **Не догма:** стек пользователя всегда уважается (принцип 1 навыка); эвристики — для обоснования предложений и предупреждений о рисках.

## Что влияет на выбор (читать из Файлов №1/№2)

| Сигнал в требованиях | Влияние на стек |
|----------------------|------------------|
| Тип проекта (backend/frontend/fullstack) | сколько треков и какие профили |
| Реалтайм (чаты, нотификации, live) | WebSocket/SSE-поддержка фреймворка; на фронте — соответствующий клиент |
| SEO / публичные страницы | SSR-фреймворк (Next/Nuxt) вместо чистого SPA → профиль `_generic_template` |
| Высокая нагрузка / низкая latency | async-стек, компилируемый язык (Go), кэширование |
| Сложный домен / транзакции | зрелый ORM, строгая слоистая архитектура |
| Команда/опыт читателя (профиль) | мейнстрим с обилием доков (легче собрать пак, легче учиться) |
| Уже заданный язык | фреймворк подбирается в рамках этого языка |

## Безопасные дефолты (если ничего не задано)

Цель дефолта — мейнстрим, активная поддержка, обилие документации (её проще исследовать для пака) и наличие курируемого профиля.

| Трек | Безопасный дефолт | Профиль |
|------|-------------------|---------|
| backend (Python) | FastAPI + SQLAlchemy(async) + Alembic + PostgreSQL + JWT | `backend_layered` |
| backend (Node) | NestJS + Prisma + PostgreSQL + JWT | `backend_layered` |
| backend (Go) | chi/gin + sqlc/GORM + PostgreSQL + JWT | `backend_layered` |
| frontend | React + Vite + TypeScript + (TanStack Query/Router или React Router) + стор по потребности | `frontend_spa` |
| frontend (альт.) | Vue 3 + Vite + TypeScript + Pinia + Vue Router | `frontend_spa` |
| fullstack | один backend-дефолт + один frontend-дефолт + общий контракт OpenAPI | оба профиля |

> Всегда добавляй TypeScript на фронте по умолчанию: типы из контракта — ключевая ценность contract-first.

## Валидация заданного стека (предупреждения, не запреты)

Проверь и, если нужно, **предупреди** (книгу всё равно строй на выборе пользователя):

- ORM/драйвер не поддерживает async, а NFR требует высокой конкуррентности → отметь усложнение.
- Чистый SPA при явном требовании SEO → предложи SSR как альтернативу, но уважай выбор.
- Библиотека устарела/архивирована (проверь по источникам) → предложи поддерживаемую замену.
- Несочетаемые версии (например, мажор фреймворка и плагина конфликтуют) → зафиксируй совместимые версии в паке.
- Экзотический фреймворк без курируемого профиля → пойдёт через `_generic_template`; предупреди, что глубина зависит от качества исследования доков.
