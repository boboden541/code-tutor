# Build-smoke gate — проверка пака сборкой до заморозки

> **Зачем.** Глубокий тест показал: все дефекты, ломавшие сборку проекта (`D1` argon2≠bcrypt, `D2` отсутствие greenlet, `D3` типизация vite/vitest, `D5` version drift фронта, `D6` `npx tsc` тянет чужой пакет, `D7` нерабочий fullstack-Docker), — **версионно/упаковочные на стыке «пак ↔ зависимости ↔ упаковка»**, а не в архитектуре или бизнес-логике. Все они ловятся автоматически, если **до заморозки** пака собрать по нему минимальный скаффолд и реально его поднять. Этот gate — обязательный этап `/tutor-stack` между сборкой пака (Этап 5) и заморозкой.
>
> **Принцип.** Пак нельзя замораживать, пока по нему не собран и не запущен минимальный скелет. «Книга воспроизводима» = «скелет из пака запускается командами из пака на версиях из пака».

## Где выполняется

`/tutor-stack`, новый под-этап **5.5 (Build-smoke gate)** — после сборки Knowledge Pack и (для fullstack) контракта, **до** простановки `STATUS: FROZEN`. Работает в скрэтч-каталоге (вне `book/`), результат в `book/` не коммитится — это эфемерная проверка пака, а не часть книги.

## Что собирается (минимальный скаффолд)

Не весь проект — **тонкий вертикальный срез на каждый трек**, ровно достаточный, чтобы упасть на тех же стыках, что и реальная книга:

### Backend-срез
1. `project_setup.md` + `00_index.md` пака → создать каталог, манифест зависимостей с **зафиксированными версиями**, установить их.
2. `first_api.md` → эндпоинт `/health`.
3. Если есть БД (`db.md`): одна модель + async-сессия + одна миграция (это вскрывает `D2` greenlet, драйвер).
4. Если есть auth (`security.md`): один вызов хэша пароля + выпуск/проверка токена (это вскрывает `D1` argon2≠bcrypt).

### Frontend-срез
1. `project_setup.md` → scaffold выбранным инструментом, установка.
2. **Сверка версий (`D5`):** после установки прочитать фактический `package.json`/lock и сверить с версиями, заявленными в `00_index.md`. Если scaffold выдал другие мажоры — либо до-зафиксировать (`npm i pkg@<pinned>`), либо обновить пак под фактические версии. Пак и установленное обязаны совпадать.
3. `first_screen.md` → один компонент рендерится.
4. `api_client.md` → http-клиент и одна типизированная DTO компилируются (если контракт к этому моменту уже сгенерирован — сверь типы с ним; если нет — достаточно, что `tsc -b`/`npm run build` проходит, полная сверка типов с контрактом — забота `/tutor-write` и `/tutor-validate`).

## Прогон проверок (через Bash)

На каждый трек — соответствующие команды **из пака** (не выдуманные):

| Проверка | Backend | Frontend |
|----------|---------|----------|
| install | `pip install -r requirements.txt` (или из пака) | `npm ci` / `npm install` |
| typecheck/build | (если типизирован) | `npm run build` — **не** голый `npx tsc` (`D6`: при отсутствии локального бинаря npx скачает чужой пакет; всегда локальный bin / `tsc -b` через `npm run build`) |
| test | `pytest` (с конфигом из `testing.md`, без варнингов — `M1`) | `npm test` (vitest/jest из пака) |
| run smoke | поднять сервер, `curl /health` → 200 | dev-сервер/preview поднимается |

## Fullstack: docker-smoke «один up — три слоя» (`D7`)

Для `fullstack` мало собрать части по отдельности — проверяется **путь до прода одной командой**:

1. По `docker.md` пака собрать: `backend/Dockerfile` + `entrypoint.sh` (**миграции в entrypoint**, не ручной шаг), `frontend/Dockerfile` + `nginx.conf` (прокси `/api/` → бэкенд), **единый корневой `docker-compose.yml` со ВСЕМИ сервисами** (db + api + web).
2. `docker compose up -d --build`.
3. Проверить **три слоя**:
   - API: `curl :<api>/health` → `{"status":"ok"}`;
   - WEB (nginx) отдаёт SPA: `curl :<web>/` → есть `<div id="root">`/`index.html`;
   - WEB→API→БД сквозняком: `curl :<web>/api/...` → валидный ответ (значит nginx проксирует, API жив, миграции применились автоматически).
4. `docker compose down`.

Критерий: **`docker compose up` без единой ручной команды поднимает все три слоя**. Если для рабочего API нужен ручной `alembic upgrade` — это дефект пака (миграции обязаны быть в entrypoint).

## Цикл само-коррекции (предел — 5 итераций)

При любом падении: **чинится ПАК** (файл `_knowledge/<tech>/*.md` и/или версии в `00_index.md`), не скаффолд. Скелет пересобирается. Типовые правки, выведенные из дефектов:

| Симптом | Правка пака |
|---------|-------------|
| `HasherNotAvailable: argon2` (`D1`) | идиома `security.md` должна соответствовать install-extra (`pwdlib[bcrypt]` → явный `BcryptHasher`, не `.recommended()`) |
| `the greenlet library is required` (`D2`) | для async-ORM в `00_index.md`/`db.md` включить рантайм-драйвер (`sqlalchemy[asyncio]`, greenlet) |
| `tsc`: поле `test` не типизируется (`D3`) | `vite.config` импортирует `defineConfig` из `vitest/config`, не из `vite` |
| фактические версии ≠ заявленным (`D5`) | до-пинить после scaffold или синхронизировать `00_index.md` с фактом |
| `npx` скачал посторонний бинарь (`D6`) | заменить на `npm run build` / локальный bin |
| `relation ... does not exist` после `up` (`D7`) | миграции в `entrypoint.sh`, единый compose со всеми сервисами |

## Выход gate

- Все срезы install + typecheck/test + run **зелёные**; для fullstack — docker-smoke «три слоя» зелёный.
- Версии в `00_index.md` совпадают с фактически установленными.
- Только теперь пак помечается `STATUS: FROZEN` + дата + версии. В индекс пака добавляется строка `BUILD_SMOKE: PASS (<дата>, <что проверено>)`.

## Деградация (нет Docker/сети)

Если в среде нет Docker или сети (headless/cron):
- выполни доступную часть (install/typecheck/test);
- пропущенное (docker-smoke) **явно** отметь в `00_index.md`: `BUILD_SMOKE: PARTIAL — docker-smoke пропущен (нет Docker в среде)`;
- не маскируй пропуск под успех. Частичный прогон лучше, чем его отсутствие, но честность о пробеле обязательна (тот же принцип, что `[NEEDS_VERIFICATION]`).
