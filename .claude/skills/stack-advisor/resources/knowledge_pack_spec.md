# Stack Knowledge Pack — спецификация

> **Что это.** База знаний под конкретный выбранный стек, собираемая `/tutor-stack` и **замораживаемая**. После заморозки `/tutor-write` пишет книгу строго против пака — это переносит на любой стек ту глубину и непротиворечивость, что раньше давали хардкод-ресурсы `book-author/resources/*.md` под FastAPI.
>
> **Где лежит.** `book/_knowledge/<tech>/*.md`. Для fullstack — отдельная подпапка на каждый стек: `book/_knowledge/<backend_tech>/`, `book/_knowledge/<frontend_tech>/` (например `book/_knowledge/fastapi/`, `book/_knowledge/react/`).

## Принципы пака

1. **Один источник истины.** Все шаги книги по данному стеку черпают конкретику (имена API, сигнатуры, идиомы, команды) отсюда. Никакой «магии из общих знаний» в обход пака.
2. **Версионирован и воспроизводим.** В индексе пака зафиксированы версии ключевых библиотек. Книга, собранная по паку, должна запускаться у читателя на тех же версиях.
3. **Честность о пробелах.** Что не удалось подтвердить источником — помечается `[NEEDS_VERIFICATION]`, а не выдумывается. Лучше отметить пробел, чем дать галлюцинацию (тот же gate, что в навыке `openapi-spec`).
4. **Заморозка.** По завершении сборки индекс помечается `STATUS: FROZEN` + дата + версии. `/tutor-write` не дописывает пак; если нужна правка — повторный `/tutor-stack`.

## Источники фактов (порядок предпочтения)

1. **Context7 MCP — основной источник** (сконфигурирован для проекта в `.mcp.json`, пред-одобрен в `.claude/settings.json`): `resolve-library-id` → `get-library-docs` для каждой ключевой библиотеки. Самый надёжный источник актуальных доков под версию. Помечай `source: context7:<resolved-id>`.
2. **WebSearch + WebFetch** — fallback, если Context7 в сессии недоступен: официальная документация, гайды «getting started», релиз-ноутсы. Помечай `source: web:<url>`.
3. **Знания модели**: только как каркас структуры; все конкретные имена/сигнатуры сверяй с источниками 1–2.

> Инструменты Context7 могут быть отложенными — найди их схемы через ToolSearch (`context7` / `resolve-library-id`) перед первым вызовом. Если сервер не поднялся (headless/cron) — переходи на WebSearch/WebFetch и явно отметь это в индексе пака.

## Обязательные темы пака

Набор тем зависит от трека. Создавай только релевантные проекту файлы; каждый файл — конкретно под выбранную технологию (не абстрактно).

### Backend-пак (`book/_knowledge/<backend_tech>/`)

| Файл | Содержание (под конкретный стек) |
|------|----------------------------------|
| `00_index.md` | список библиотек+версии, источники, STATUS/дата, карта файлов пака |
| `architecture.md` | как слои профиля ложатся на каталоги/файлы этого фреймворка |
| `project_setup.md` | scaffold, менеджер пакетов, структура, запуск dev-сервера |
| `first_api.md` | минимальный эндпоинт `/health`, как фреймворк регистрирует маршруты |
| `db.md` *(если есть БД)* | подключение, ORM/драйвер, сессии, миграции |
| `security.md` *(если есть auth)* | хэширование паролей, токены/сессии, защита роутов |
| `testing.md` | тест-раннер, клиент для запросов, фикстуры |
| `docker.md` *(если контейнеризуется)* | Dockerfile, compose под этот стек |
| `glossary.md` | термины стека для «Разбора кода» (DI, декоратор, ORM, миграция…) |
| `gotchas.md` | частые ошибки/ловушки именно этого стека |

### Frontend-пак (`book/_knowledge/<frontend_tech>/`)

| Файл | Содержание (под конкретный стек) |
|------|----------------------------------|
| `00_index.md` | библиотеки+версии, источники, STATUS/дата, карта файлов |
| `architecture.md` | как слои профиля (api_client/store/hooks/components) ложатся на структуру |
| `project_setup.md` | scaffold (Vite/CLI), менеджер пакетов, dev-сервер |
| `first_screen.md` | первый компонент/экран, точка входа, как идёт рендер |
| `routing.md` | роутер стека, определение маршрутов |
| `state.md` | управление состоянием (стор/сигналы/контекст) |
| `api_client.md` | http-клиент, генерация/написание типов из контракта OpenAPI |
| `testing.md` | тест-раннер + библиотека тестирования компонентов |
| `glossary.md` | термины (компонент, состояние, хук/композабл, реактивность, проп…) |
| `gotchas.md` | частые ошибки этого фреймворка |

> Темы деплоя/CI общие по духу, но команды — стек-специфичны: добавляй `deploy.md`/`ci.md` в пак того трека, чьи артефакты деплоятся.

## Формат файла пака

```markdown
# <Тема> — <Технология вер. X.Y>
> source: context7:<lib> | web:<url>
> verified: <дата>

## <Подтема>
<конкретные факты: имена API, сигнатуры, идиомы, минимальные примеры>
<непроверенное → [NEEDS_VERIFICATION]: что проверить>
```

## Процедура сборки (для `/tutor-stack`, Этап 4)

1. Определи список ключевых библиотек стека (из решения по стеку).
2. Для каждой — добудь доки (Context7 → web).
3. Заполни обязательные темы пака под трек (см. таблицы), под конкретные версии.
4. Прогони само-проверку: покрыты ли темы; нет ли противоречий между файлами; зафиксированы ли версии.
5. Запиши `00_index.md` с `STATUS: FROZEN`, датой, списком версий и источников.
6. После заморозки пак — read-only для `/tutor-write`.
