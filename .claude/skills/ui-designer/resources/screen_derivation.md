# Вывод экранов из контракта и требований

> Как из API-контракта + требований получить полный, непротиворечивый список экранов с матрицей покрытия. Цель — не пропустить ни одной функции (каждый эндпоинт достижим) и ни одного ролевого сценария.

## §0. Сначала — суть (для эстетики)
До экранов назови: **один конкретный предмет** продукта, **аудиторию**, **единственную задачу** главного экрана. Из этого выводится визуальный язык (палитра/типографика/фирменный элемент) — см. навык `frontend-design`. Пример: «маркетплейс подарков; покупатели и продавцы-флористы/кондитеры; задача каталога — быстро выбрать и заказать подарок с гарантией ‘фото до доставки’» → тёплая, праздничная, доверительная эстетика.

## §1. Источники
- **Контракт** `book/contracts/<domain>.yaml`: список операций (метод+путь+operationId), схемы запросов/ответов с ограничениями (для валидации форм).
- **Требования** `requirements.md`: роли (UR), функциональные сценарии (FR), машина состояний домена.

## §2. Алгоритм
1. Выпиши **все операции контракта** в таблицу (метод, путь, что делает, какая роль вызывает).
2. Выпиши **роли** и для каждой — её happy-path (последовательность действий «от входа до цели»).
3. Разложи операции по экранам так, чтобы:
   - **чтения** (`GET`) формировали экраны просмотра (списки, детали);
   - **мутации** (`POST/PUT/DELETE`) стали действиями на экранах (кнопки/формы);
   - один экран ⊇ связная группа операций одной задачи.
4. Добавь служебные экраны: auth (register/login), каркас/навигация, экраны пустых/ошибочных состояний (как состояния, не отдельные роуты).
5. Зафиксируй навигацию: входные точки на роль, переходы, редиректы после действий, role-gating.

## §3. Матрица покрытия (обязательна)
Построй таблицу «экран × эндпоинт». Проверки:
- **Каждый эндпоинт** контракта стоит хотя бы в одной ячейке (нет непотреблённых → иначе функция недостижима, дефект `endpoint-unconsumed`).
- **Каждый ролевой флоу** проходим по экранам без разрывов (нет «тупиков»).
- **Каждая мутация** привязана к конкретному действию UI.

Пример (фрагмент):

| Экран | GET-чтения | Мутации (действия) |
|-------|-----------|--------------------|
| catalog | `GET /products` | — |
| buyer_order_detail | `GET /orders/{id}` | pay, cancel, approve-photo, reject-photo |
| seller_order_detail | `GET /orders/{id}` | accept, photo (upload URL), deliver |

## §4. Формы и валидация (из схем контракта)
Для каждого экрана с вводом возьми ограничения из схемы запроса: `required`, `minLength`/`maxLength`, `gt`/`ge`, enum, формат (дата/email/URL). Перенеси их в `input_forms` как клиентскую валидацию — не строже и не слабее контракта. Деталь: если поле «фото» в контракте — это `string`-URL (а не `binary`), форма вводит URL, а не файл.

## §5. Состояния (из реальности данных)
Для каждого экрана с данными опиши loading/empty/error; для каждой мутации — pending/success(invalidation)/failure(человекочитаемый текст по коду). Это и есть «нет белых страниц».

## §6. Результат
Список экранов (каждый по `screen_spec_schema.md`) + матрица покрытия → в `design_spec.md`. Имена экранов (`screen_id`) станут именами шагов книги, поэтому snake_case латиницей.
