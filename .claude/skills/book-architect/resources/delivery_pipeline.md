# Скелет конвейера разработки (порядок глав книги)

> Опорный скелет для `book-architect`. **Стек-агностичен.** Конкретный порядок глав и набор слоёв фичи берётся из **Architecture Profile** (`profiles/`), резолвленного шагом `/tutor-stack`, а не из хардкода под FastAPI.
> Адаптируй под конкретный проект (Файлы №1/№2/№3): добавляй/убирай секции, но сохраняй порядок зависимостей.

## Как читать этот документ

1. `/tutor-stack` определил **тип проекта** (backend/frontend/fullstack) и **профиль(и)** архитектуры.
2. Для одиночного трека — бери `chapter_skeleton` из соответствующего профиля (`backend_layered` / `frontend_spa` / заполненный `_generic_template`).
3. Для **fullstack** — собери книгу из двух частей + контракта по разделу «Fullstack» ниже.

## Общий порядок глав (любой трек)

| # | Глава | Что осваивается | Результат главы |
|---|-------|------------------|------------------|
| 0 | prerequisites *(если junior/чистая машина)* | установка инструментов стека под ОС читателя | окружение готово, версии проверены |
| 1 | starting_project | репозиторий, scaffold, структура по профилю, первый запускаемый артефакт, первый тест | приложение поднимается локально, тест зелёный |
| 2 | foundation *(если нужно)* | бэк: БД/миграции/auth; фронт: роутинг/layout/http-клиент | каркас, на который вешаются фичи |
| 3.. | features (по числу доменов/экранов) | повторение слоёв профиля для каждой фичи | функционал MVP |
| N | delivery | CI, секреты, сборка, прод-сервер, деплой, smoke-тест | продукт доступен/развёрнут |

## Правила декомпозиции на секции и шаги

1. **Глава → секции по фичам/этапам**, секция → шаги **по слоям из профиля** (а не из хардкода).
2. **Итеративный git-flow (обязательно).** Каждая секция-фича обрамляется:
   - первый шаг — `start_feature_branch.md` (ветка `feat/<name>` от `main`);
   - шаги по слоям профиля;
   - последний шаг — `commit_push_open_mr.md` (тесты → commit → push → MR → merge в `main`).
   См. `book-author/.../git_workflow.md`.
3. **Первая фича трека: один слой = один шаг.** Порядок шагов-файлов = `layers` из профиля. Запрещено сворачивать слои первой фичи в один `<f>_feature.md`.
4. **Вторая фича:** слои сгруппированы в 1–2 шага. **Третья и далее:** «по образцу» со ссылкой на 1-ю фичу.
5. Инфраструктурные шаги (БД, миграции, сборщик, CI) идут до шагов, которые от них зависят; их секции — через ветку `chore/...`.
6. Глава 1 «старт проекта» работает в `main` и содержит шаг-памятку `how_we_work_git_flow.md`; со второй главы каждая фича — отдельная ветка.
7. **Глава 0** (`chapter_0_prerequisites`) генерируется первой, если профиль читателя в Файле №2 — junior/чистая машина (база — `prerequisites.md` в нужном паке).
8. **Обзор потока:** первая доменная глава открывается шагом `mental_model_ref` профиля (бэк → `request_flow_overview`; фронт → `data_flow_overview`).
9. **Рекап:** каждая глава завершается шагом `recap_and_state.md` («Что мы поняли» + полный `state_snapshot_file` профиля + дерево проекта).
10. Каждая глава завершается **проверяемым результатом**; каждая фича — влитой в `main` веткой.
11. Обязательные сквозные темы, даже если пользователь не указал: безопасность, тестирование, деплой, итеративная работа с ветками.

## Fullstack: контракт-first срезы по фичам

Для `fullstack` книга строится так (две части + общий контракт), порядок выбран пользователем: **контракт-first + вертикальные срезы по фичам**.

### Общая форма книги

```
part_1_backend/   — бэкенд-трек по профилю backend_layered
part_2_frontend/  — фронтенд-трек по профилю frontend_spa
contracts/        — OpenAPI, единый источник правды для обеих частей
```

### Порядок изучения

1. **Общий фундамент сначала** (внутри каждой части): глава 0 (окружение для обоих стеков), старт обеих кодовых баз (бэк-scaffold, фронт-scaffold), первый запускаемый артефакт каждой стороны.
2. **Затем — срез на каждую фичу**, повторяемый цикл:
   1. `define_api_contract` — зафиксировать/уточнить контракт фичи в `contracts/<domain>.yaml` (служебный шаг; OpenAPI как «договор»).
   2. **Бэкенд-реализация фичи** — по слоям профиля `backend_layered`, сверяясь с контрактом (часть 1).
   3. **Фронтенд-потребление фичи** — по слоям профиля `frontend_spa`, типы api_client из контракта (часть 2).
   4. `wire_end_to_end` — сквозная проверка: фронт реально дёргает бэк, фича работает целиком (служебный шаг).
3. **Доставка** — CI и деплой обеих частей (могут быть отдельные секции в каждой части или общая глава delivery).

> Где разместить шаги среза. Контракт — в `contracts/`. Бэк-шаги среза — в соответствующей главе `part_1_backend/`, фронт-шаги — в `part_2_frontend/`. Шаг `wire_end_to_end` кладётся в фронтенд-часть (она замыкает цикл), но в его «Предусловиях» явно ссылается на бэкенд-шаги среза.

> Почему контракт-first. Контракт (OpenAPI) фиксирует шов между командами/частями до кода: бэк обязуется отдавать ровно это, фронт строит типы ровно под это. Это убирает класс ошибок «фронт ждал одно, бэк отдал другое» и делает обе части независимо проверяемыми. Контракт генерирует `/tutor-stack` (см. `stack-advisor/.../contract_first.md`).

## Убывающая детализация (полнота кода по фичам)

Кривая сложности плавная, без обрыва (в КАЖДОМ треке отдельно):
- **1-я фича** — по слою на шаг, полный копируемый код + «Разбор кода» в каждом шаге.
- **2-я фича** — слои сгруппированы, код полный, разбираются только новые/отличающиеся места со ссылкой на 1-ю.
- **3-я+** — «по образцу»: только отличия + обязательная ссылка на слой 1-й фичи.
Запрещено давать первую фичу «по образцу» — у новичка ещё нет образца.

## Сопоставление с требованиями (Файл №2)

- Каждый FR → минимум один шаг в главе домена (в соответствующем треке).
- Каждый IR (эндпоинт/контракт) → запись в `contracts/` + шаг реализации (бэк) + шаг потребления (фронт, если есть).
- Каждый NFR → отражается в шаге (security/перформанс/деплой) или в критерии «результат шага».
- Непокрытых пунктов Файла №2 быть не должно — это проверит `/tutor-validate`.
