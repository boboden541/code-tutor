# Скелет конвейера разработки (порядок глав книги)

> Опорный скелет для `book-architect`. Это рекомендованный порядок глав «от пустой папки до прод-стенда».
> Адаптируй под конкретный проект (Файлы №1/№2): добавляй/убирай секции, но сохраняй порядок зависимостей.

## Порядок глав

| # | Глава (chapter) | Что осваивается | Результат главы |
|---|-----------------|------------------|------------------|
| 0 | prerequisites *(если junior/чистая машина)* | установка Python, Docker, git, SSH-ключ под ОС читателя | окружение готово: версии проверены, `docker` и `git` работают |
| 1 | starting_project | репозиторий, структура чистой архитектуры, первое API, Docker, первый тест | приложение поднимается локально и в контейнере, тест зелёный |
| 2 | users_and_auth (или домен проекта) | БД, миграции, сущности, репозитории, usecase, роутеры, безопасность, тесты | работающее API ключевых сущностей с аутентификацией |
| 3 | features (по числу доменов) | повторение слоёв чистой архитектуры для остальных сущностей | полный функционал MVP |
| 4 | delivery | GitLab CI, секреты, подготовка прод-сервера, деплой, smoke-тест | продукт доступен в интернете на проде |

## Правила декомпозиции на секции и шаги

1. **Глава → секции по фичам/этапам**, секция → шаги по слоям чистой архитектуры.
2. **Итеративный git-flow (обязательно).** Каждая секция-фича обрамляется:
   - первый шаг — `start_feature_branch.md` (ветка `feat/<name>` от `main`);
   - шаги по слоям (см. п.3);
   - последний шаг — `commit_push_open_mr.md` (тесты → commit → push → MR → merge в `main`).
   Полный цикл фичи: `ветка → реализация по слоям → тесты → commit/push → MR → merge`. См. `git_workflow.md`.
3. **Первая фича: один слой = один шаг.** Порядок шагов-файлов: `<f>_schema → <f>_domain → <f>_repository_interface → <f>_repository_impl → <f>_migration → <f>_usecase → <f>_router → <f>_test` (+ `<f>_security`, если нужно). Запрещено сворачивать слои первой фичи в один `<f>_feature.md`.
4. **Вторая фича:** слои сгруппированы в 1–2 шага. **Третья и далее:** «по образцу» со ссылкой на 1-ю фичу (см. `SKILL.md`, `## 4a`).
5. Инфраструктурные шаги (БД, миграции, Docker, CI) идут до шагов, которые от них зависят; их секции тоже идут через ветку (`chore/...`).
6. Глава 1 «старт проекта» работает в `main` и содержит шаг-памятку `how_we_work_git_flow.md`, объясняющий итеративную модель; со второй главы каждая фича — отдельная ветка.
7. **Глава 0** (`chapter_0_prerequisites`) генерируется первой, если профиль читателя в Файле №2 — junior/чистая машина (база — `prerequisites.md`).
8. **Обзор потока:** первая доменная глава открывается шагом `request_flow_overview.md` (база — `mental_models.md`).
9. **Рекап:** каждая глава завершается шагом `recap_and_state.md` («Что мы поняли» + полный `requirements.txt` + дерево проекта).
10. Каждая глава завершается **проверяемым результатом** (см. колонку «Результат главы»); каждая фича — влитой в `main` веткой.
11. Обязательные сквозные темы, даже если пользователь не указал: безопасность (хэширование, секреты), тестирование, деплой, итеративная работа с ветками.

## Убывающая детализация (полнота кода по фичам)

Кривая сложности плавная, без обрыва:
- **1-я фича** — по слою на шаг, полный копируемый код + «Разбор кода» в каждом шаге.
- **2-я фича** — слои сгруппированы, код полный, разбираются только новые/отличающиеся места со ссылкой на 1-ю.
- **3-я+** — «по образцу»: только отличия + обязательная ссылка на слой 1-й фичи.
Запрещено давать первую фичу «по образцу» — у новичка ещё нет образца.

## Сопоставление с требованиями (Файл №2)

- Каждый FR → минимум один шаг в главе домена.
- Каждый IR (эндпоинт/контракт) → шаг router + шаг test.
- Каждый NFR → отражается в шаге (security/перформанс/деплой) или в критерии «результат шага».
- Непокрытых пунктов Файла №2 быть не должно — это проверит `/tutor-validate`.
