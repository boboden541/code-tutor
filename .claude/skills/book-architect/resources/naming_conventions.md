# Naming Conventions — code-tutor

> Единый словарь терминов и правил именования для всех скилов и команд продукта `code-tutor`.
> Источник истины. Все скилы (`project-brief`, `requirements-interview`, `stack-advisor`, `book-architect`, `book-author`) и команды (`/tutor-*`) обязаны использовать эти термины без синонимов.
>
> **Что такое code-tutor:** генератор обучающей книги по разработке проекта «от пустой папки до прод-стенда». Стек **не зашит**: проект может быть backend, frontend или fullstack на любом языке/фреймворке (Python/Go/Java/Node + React/Vue/…). Глубина книги обеспечивается не хардкодом ресурсов под FastAPI, а механизмом **Architecture Profile** (профиль архитектуры на технологию) + **Stack Knowledge Pack** (замораживаемая база знаний под выбранный стек).

## 1. Артефакты конвейера

| Термин | Файл / каталог (по умолчанию) | Кто создаёт | Описание |
|--------|-------------------------------|-------------|----------|
| Файл №1 / project brief | `project_brief.md` | `/tutor-brief` | Идея проекта, тип (backend/frontend/fullstack), домен, сценарии, профиль читателя. **Стек — опционален** |
| Файл №2 / requirements | `requirements.md` | `/tutor-requirements` | Опросник по 4 категориям требований (стек-агностично) |
| Файл №3 / stack decision | `stack_decision.md` | `/tutor-stack` | Утверждённый стек + обоснование + резолв Architecture Profile(ей) |
| API-контракт / contract | `book/contracts/<domain>.yaml` | `/tutor-stack` | OpenAPI 3.0.3 — шов между фронтом и бэком (для fullstack/backend) |
| Файл №4 / design spec | `design_spec.md` | `/tutor-design` | Спецификация экранов + дизайн-система (для frontend/fullstack); источник UI для плана и записи |
| Knowledge Pack / база знаний стека | `book/_knowledge/<tech>/*.md` | `/tutor-stack` | Замороженная база знаний под выбранные технологии (единый источник истины для записи) |
| План книги / outline | `book/outline.md` | `/tutor-plan` | Машиночитаемый план книги с чек-боксами |
| Книга / book | `book/` | `/tutor-write` | Дерево частей/глав/секций/шагов |
| Отчёт валидации | (выводится в чат) | `/tutor-validate` | Список расхождений + вердикт PASS/FAIL |

## 2. Тип проекта (управляет структурой книги)

Фиксируется в Файле №1 (поле «Тип проекта») и переносится в Файл №3.

| Тип | Что строит книга | Треки книги |
|-----|------------------|-------------|
| `backend` | Серверное приложение/API | один трек (`book/backend/**` или просто `book/**`) |
| `frontend` | Клиентское SPA/приложение | один трек (`book/frontend/**` или просто `book/**`) |
| `fullstack` | Бэкенд + фронтенд + контракт между ними | два трека: `book/backend/**` и `book/frontend/**` + общий `book/contracts/**` |

Для **одиночного** трека (backend ИЛИ frontend) уровень `part_*` не используется — главы лежат прямо в `book/`. Для **fullstack** вводится уровень частей (см. §3a).

## 3. Структура книги (одиночный трек)

```
book/
├── outline.md
├── _knowledge/               # Stack Knowledge Pack (заморожен, источник истины для записи)
│   └── <tech>/*.md
├── contracts/                # API-контракты OpenAPI (если есть API)
│   └── <domain>.yaml
├── chapter_1_{name}/
│   ├── section_1_{name}/
│   │   ├── 01_{step_name}.md
│   │   └── 02_{step_name}.md
│   └── section_2_{name}/
│       └── 01_{step_name}.md
└── chapter_2_{name}/
    └── ...
```

| Уровень | Термин | Шаблон каталога/файла | Пример |
|---------|--------|------------------------|--------|
| 1 | Глава (chapter) | `chapter_<N>_<snake_name>/` | `chapter_1_starting_project/` |
| 2 | Секция (section) | `section_<M>_<snake_name>/` | `section_2_make_first_api/` |
| 3 | Шаг (step) | `<KK>_<snake_name>.md` | `01_create_first_api.md` |

## 3a. Структура книги (fullstack — два трека)

Для `fullstack` вводится верхний уровень **части (part)**: один трек = одна часть. Книга строится **контракт-first срезами по фичам** (см. `delivery_pipeline.md`): контракт → бэкенд-реализация → фронтенд-потребление → сквозная проверка.

```
book/
├── outline.md
├── _knowledge/
│   ├── <backend_tech>/*.md
│   └── <frontend_tech>/*.md
├── contracts/
│   └── <domain>.yaml          # единый источник правды для обеих частей
├── part_1_backend/
│   ├── chapter_1_.../section_1_.../01_....md
│   └── ...
└── part_2_frontend/
    ├── chapter_1_.../section_1_.../01_....md
    └── ...
```

| Уровень | Термин | Шаблон каталога | Пример |
|---------|--------|------------------|--------|
| 0 | Часть (part) | `part_<P>_<track>/` | `part_1_backend/`, `part_2_frontend/` |
| 1 | Глава (chapter) | `chapter_<N>_<snake_name>/` | `chapter_1_setup_tooling/` |
| 2 | Секция (section) | `section_<M>_<snake_name>/` | `section_1_incomes/` |
| 3 | Шаг (step) | `<KK>_<snake_name>.md` | `02_incomes_api_client.md` |

`<P>` частей — `1` (backend), `2` (frontend). Нумерация глав `<N>` — **сквозная внутри своей части** (в каждой части отсчёт с 1, либо с 0 если есть глава 0).

## 4. Правила именования

1. `<N>` глав — целые числа, **сквозные без пропусков и дублей** внутри своего трека/части. Нумерация с `1`; допускается спец-глава `chapter_0_prerequisites` — тогда отсчёт с `0`. `<M>` секций — с `1`. `<P>` частей — с `1`.
2. **Файлы-шаги ОБЯЗАТЕЛЬНО начинаются с 2-значного префикса порядка** `<KK>_` (`01_`, `02_`, …): сквозной, с `01`, без пропусков и дублей **внутри секции**. Префикс делает порядок шагов видимым прямо в списке файлов.
3. `<name>` — `snake_case`, латиница, глагол+объект, отражает действие: `create_user_api`, `setup_vite`, `deploy_to_prod`. **Кириллица и пробелы в путях книги запрещены** (даже для русскоязычного содержимого файлов).
4. Имена файлов-шагов уникальны в пределах своей секции.
5. **«Шаг N из M в секции»** в заголовке файла — НЕ пишется вручную: `N` = число из префикса `<KK>`, `M` = количество шагов в секции. Совпадение проверяется валидатором.
6. Все ссылки между файлами книги — **относительные** от корня проекта (`book/...`). Абсолютные пути запрещены.
7. **Служебные шаги** (не привязаны к слою, префикс `<KK>_` обязателен): `how_we_work_git_flow`, `request_flow_overview`, `data_flow_overview`, `recap_and_state`, `start_feature_branch`, `commit_push_open_mr`, `setup_environment`, `define_api_contract`, `wire_end_to_end`, `design_system`, `app_shell_and_routing`, `protected_route`.

## 5. Формат строки `outline.md` (контракт для `/tutor-write`)

Каждый шаг — одна строка:

```
- [ ] <относительный_путь_к_шагу>.md — <одна строка назначения>
```

- `[ ]` — шаг не написан; `[x]` — шаг написан и принят.
- Регэксп-контракт: `^- \[( |x)\] (\S+\.md) — (.+)$` (путь шага оканчивается на `<KK>_<name>.md`).
- Заголовки частей (только fullstack) в outline: `# part_<P>_<track>`.
- Заголовки глав в outline: `## chapter_<N>_<name>`.
- Внутри секции шаги перечислены **по возрастанию префикса** `<KK>` — порядок в outline = порядок префиксов = порядок файлов в папке.
- Тире-разделитель назначения — символ `—` (em dash, U+2014).

## 6. Терминология стека и архитектуры

| Термин | Описание |
|--------|----------|
| Стек (stack) | Конкретные язык/фреймворк/библиотеки. Берётся из Файла №1 (если задан пользователем) или предлагается `/tutor-stack` |
| Architecture Profile | Декларативный профиль на технологию: порядок слоёв фичи, скелет глав, файл-снимок состояния, тест-раннер, билд-тул, модель деплоя (см. `profiles/_profile_schema.md`) |
| Stack Knowledge Pack | База знаний под выбранный стек (`book/_knowledge/<tech>/`), собранная `/tutor-stack` и **замороженная**: при записи книги — единственный источник истины, как раньше были `book-author/resources/*.md` под FastAPI |
| Файл-снимок состояния | Манифест зависимостей трека: `requirements.txt`/`pyproject.toml` (Python), `package.json` (Node/JS), `go.mod` (Go), `pom.xml`/`build.gradle` (Java). Приводится целиком в каждом `recap_and_state` |
| Слой фичи | Шаг по слою архитектуры; набор и порядок слоёв задаёт Architecture Profile, а не хардкод (бэк: `schema→domain→repo→usecase→router→test`; фронт-SPA: `api_client→store→hooks→components→styles→test`) |
| Файл №4 / design spec | Спецификация экранов (маршрут, роль, вызовы API, формы+валидация, парсинг, состояния, навигация) + дизайн-система (токены/компоненты/тон). Создаётся `/tutor-design`, потребляется `/tutor-plan` и `/tutor-write` для frontend/fullstack |

## 7. Запрещено

- Менять термины на синонимы («раздел» вместо «секция», «урок» вместо «шаг»).
- Использовать пробелы или кириллицу в именах каталогов/файлов книги.
- **Хардкодить технологический стек** в скилах/ресурсах ядра: стек берётся из Файла №1 или Файла №3, конкретика технологии — из Architecture Profile и Stack Knowledge Pack.
- Писать книгу против незамороженного Knowledge Pack (сначала `/tutor-stack` собирает и фиксирует пак, потом `/tutor-write`).
