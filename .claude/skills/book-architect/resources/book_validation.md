# Чек-лист валидации книги (уровень книги, межсекционная целостность)

> Используется командой `/tutor-validate`. Проверяет то, что не видно отдельной секции.
> Валидатор только читает — файлы книги не модифицируются.

## 1. Соответствие плану и диску

- [ ] Каждый шаг из `outline.md` существует на диске по точному пути
- [ ] Каждый шаг, существующий на диске, отмечен в плане как `[x]`
- [ ] [CRITICAL] Нет шагов `[x]` без файла на диске → `missing`
- [ ] Нет файлов `book/**/*.md` (кроме `outline.md`), отсутствующих в плане → `orphan`

## 2. Формат плана

- [ ] Все строки шагов соответствуют регэкспу `^- \[( |x)\] (\S+\.md) — (.+)$`
- [ ] Заголовки глав в формате `## chapter_<N>_<name>`; для fullstack — части `# part_<P>_<track>`
- [ ] Все пути — относительные от корня (`book/...`), без абсолютных

## 3. Нумерация и структура

- [ ] Главы пронумерованы сквозно без пропусков и дублей; старт с `1`, либо с `0` если есть `chapter_0_prerequisites` → иначе `numbering`
- [ ] Секции внутри главы пронумерованы сквозно с 1
- [ ] Имена каталогов/файлов — `snake_case`, латиница (см. `naming_conventions.md`)
- [ ] Каждый файл-шаг имеет 2-значный префикс `<KK>_`; префиксы в секции сквозные с `01`, без пропусков и дублей → иначе `step-order`
- [ ] Заголовок «Шаг N из M в секции» совпадает: `N` = префикс файла, `M` = число шагов в секции → иначе `step-count-mismatch`

## 4. Перекрёстные ссылки

- [ ] Все относительные ссылки внутри шагов ведут на существующие файлы → иначе `broken-link`
- [ ] Ссылки «предусловия» и «что дальше» в шагах корректны

## 5. Покрытие требований (Файл №2)

- [ ] Каждый пункт FR/UR/IR/NFR из `requirements.md` покрыт хотя бы одной секцией → иначе `uncovered-requirement`
- [ ] Обязательные сквозные темы присутствуют: безопасность, тесты, деплой на прод

## 6. Связность пути «от пустой папки до прода»

- [ ] Первая глава начинается с создания проекта/scaffold по профилю
- [ ] Есть глава деплоя, завершающаяся проверяемым результатом (бэк: `/health` из интернета; фронт: открытие собранного приложения в браузере)
- [ ] Порядок глав не нарушает зависимости (БД до использования БД, scaffold до запуска, контракт до реализации фичи)
- [ ] **Единый `docker-compose.yml`** со всеми сервисами (fullstack: db+api+web), поднимаемый одной командой → иначе `no-single-compose`
- [ ] Миграции применяются **автоматически** (entrypoint бэкенда), а не ручным шагом деплоя; шаг деплоя не содержит ручного `alembic upgrade`/аналога → иначе `manual-migration`
- [ ] Есть финальный smoke «один up — три слоя» (API `/health` + фронт-статика + сквозняк фронт→API→БД) → иначе `missing-prod-smoke`
- [ ] Есть шаг создания `README.md` проекта (сервисы + как поднять) → иначе `missing-readme`

## 7. Итеративный git-flow (одна фича = одна ветка = один MR)

- [ ] Есть шаг-памятка `how_we_work_git_flow` в главе 1
- [ ] Каждая секция-фича (со 2-й главы) начинается шагом `start_feature_branch` → иначе `missing-branch-step`
- [ ] Каждая секция-фича заканчивается шагом `commit_push_open_mr` → иначе `missing-mr-step`
- [ ] **Обрамление применено в КАЖДОЙ части** (для fullstack — и `part_1_backend/`, и `part_2_frontend/`): фронтенд-фичи (`auth_ui`/`catalog`/…) тоже обрамлены, не только бэкенд → иначе `missing-branch-step`/`missing-mr-step`
- [ ] Обычные шаги реализации не дублируют создание ветки (ветка одна на секцию)

## 8. Полнота и понятность кода

- [ ] В шаге нет `...`, `# (фрагмент)`, `(фрагмент)` или «по образцу» **без ссылки** на шаг с полным примером → иначе `placeholder`
  - **Исключение:** `...` как валидный синтаксис языка не считается плейсхолдером — тело `Protocol`/`@abstractmethod`/заглушки интерфейса в Python (`def m(self) -> int: ...`), пустой блок в TS (`{}`), `pass`. Плейсхолдер — это `...` *вместо опущенного кода*, а не язык.
- [ ] После каждого нетривиального блока кода есть раздел «Разбор кода» → иначе `unexplained-code`
- [ ] В коде шага не используются неопределённые в нём и нигде ранее имена (`create_uc`, `session` без сборки/`Depends`) → иначе `orphan-symbol`

## 9. Закрепление и состояние

- [ ] Каждая глава заканчивается шагом `recap_and_state` → иначе `missing-recap`
- [ ] Первая доменная глава трека открывается обзором `mental_model_ref` профиля (бэк → `request_flow_overview`; фронт → `data_flow_overview`) → иначе `missing-overview`
- [ ] Каждая зависимость из шагов (установка/правки манифеста) присутствует в финальном **файле-снимке состояния** профиля (`requirements.txt`/`package.json`/`go.mod`/…) в recap → иначе `deps-drift`

## 10. Стек, профиль, пак, контракт (стек-aware)

- [ ] Существует Файл №3 (`stack_decision.md`) с типом проекта, стеком(ами) и резолвленным профилем(ями)
- [ ] Knowledge Pack (`book/_knowledge/<tech>/`) существует на каждый трек и помечен `STATUS: FROZEN` → иначе `pack-not-frozen`
- [ ] Если профиль `_generic_template` — критичные поля (layers, test_runner, build) заполнены, без `[уточнить]` → иначе `profile-incomplete`
- [ ] Слои фичи в шагах соответствуют `layers` профиля (порядок и состав) → иначе `profile-layer-mismatch`
- [ ] Для fullstack: есть части `part_1_backend/` и `part_2_frontend/`; для каждой фичи есть бэк-реализация и фронт-потребление → иначе `part-structure`
- [ ] Для API/fullstack: пути/коды/формы в шагах соответствуют `book/contracts/<domain>.yaml`; типы фронта соответствуют схемам контракта → иначе `contract-drift`
- [ ] В шагах нет конкретики (версии/имена API), противоречащей Knowledge Pack → иначе `pack-drift`

## 11. UI / дизайн (только если есть Файл №4 `design_spec.md` и фронт-трек)

> Все проверки этого раздела условны: применяются, только если проект frontend/fullstack и существует `design_spec.md`. Backend-only книги раздел пропускают (остаются PASS). Эвристичные проверки (`unstyled-screen`/`design-token-drift`/`unwired-nav`) при сомнении — предупреждение, а не жёсткий FAIL.

- [ ] Для frontend/fullstack существует `design_spec.md` (Файл №4) → иначе `design-spec-missing`
- [ ] Каждый экран из `design_spec.md` имеет ≥1 шаг в книге (по `screen_id`) → иначе `missing-screen`
- [ ] Есть шаг каркаса (`app_shell_and_routing`) и шаг дизайн-системы (`design_system`) → иначе `missing-app-shell`
- [ ] Каждый навигационный переход из spec реализован (ссылка/редирект на целевой маршрут) → иначе `unwired-nav`
- [ ] Экраны стилизованы через токены/CSS Modules; нет нестилизованного экрана-заглушки (`<h1>` вместо реализации, «TODO») → иначе `unstyled-screen`
- [ ] Компоненты не хардкодят цвета/отступы вместо токенов → иначе `design-token-drift`
- [ ] Каждый эндпоинт контракта потребляется хотя бы одним экраном → иначе `endpoint-unconsumed`

---

## Вердикт

- **PASS** — ни одного расхождения по разделам 1–11.
- **FAIL** — есть хотя бы одно расхождение. В отчёте: `Тип | Путь | Детали`.

## Типы расхождений

| Тип | Значение |
|-----|----------|
| `missing` | шаг плана отсутствует на диске / не отмечен `[x]` |
| `orphan` | файл книги отсутствует в плане |
| `broken-link` | ссылка ведёт в несуществующий файл |
| `numbering` | пропуск/дубль нумерации глав/секций |
| `uncovered-requirement` | пункт Файла №2 не покрыт секцией |
| `missing-branch-step` | секция-фича без `start_feature_branch` |
| `missing-mr-step` | секция-фича без `commit_push_open_mr` |
| `placeholder` | `...`/«фрагмент»/«по образцу» без ссылки на полный пример |
| `unexplained-code` | блок кода без раздела «Разбор кода» |
| `orphan-symbol` | использовано имя, не определённое и не объяснённое в шаге |
| `missing-recap` | глава без шага `recap_and_state` |
| `missing-overview` | первая доменная глава трека без обзора `mental_model_ref` профиля |
| `deps-drift` | зависимость из шагов отсутствует в финальном файле-снимке состояния (recap) |
| `step-order` | файл-шаг без префикса `<KK>_` либо префиксы в секции с пропуском/дублем |
| `step-count-mismatch` | «Шаг N из M» не совпадает с префиксом файла и числом шагов в секции |
| `pack-not-frozen` | Knowledge Pack отсутствует или не помечен `STATUS: FROZEN` |
| `profile-incomplete` | `_generic_template` с незаполненными критичными полями |
| `profile-layer-mismatch` | слои фичи в шагах не соответствуют `layers` профиля |
| `part-structure` | fullstack без частей backend/frontend или фича без одной из сторон |
| `contract-drift` | пути/коды/формы/типы в шагах не соответствуют контракту OpenAPI |
| `pack-drift` | конкретика в шаге противоречит замороженному Knowledge Pack |
| `no-single-compose` | нет единого `docker-compose.yml` со всеми сервисами (fullstack: db+api+web) |
| `manual-migration` | миграции применяются ручным шагом, а не авто-entrypoint'ом |
| `missing-prod-smoke` | нет финального smoke «один up — три слоя» |
| `missing-readme` | нет шага создания `README.md` проекта |
| `design-spec-missing` | frontend/fullstack без Файла №4 `design_spec.md` |
| `missing-screen` | экран из `design_spec.md` не имеет шага в книге |
| `missing-app-shell` | нет шага каркаса/роутинга или дизайн-системы (фронт) |
| `unwired-nav` | переход из `design_spec` не реализован (нет ссылки/редиректа) |
| `unstyled-screen` | экран без стилей/с заглушкой вместо реализации |
| `design-token-drift` | хардкод цветов/отступов вместо токенов дизайн-системы |
| `endpoint-unconsumed` | эндпоинт контракта не потребляется ни одним экраном |
