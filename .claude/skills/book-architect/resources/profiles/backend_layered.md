# Architecture Profile: backend_layered

> Профиль для серверных приложений/API со слоистой («чистой») архитектурой. Подходит FastAPI, Django REST, Spring Boot, Express/NestJS, Go (chi/gin), Laravel и аналогам. Конкретные имена файлов/библиотек берутся из Stack Knowledge Pack (`book/_knowledge/<backend_tech>/`), структура — отсюда.

- **track:** `backend`
- **state_snapshot_file:** манифест зависимостей стека — `requirements.txt`/`pyproject.toml` (Python), `package.json` (Node), `go.mod` (Go), `pom.xml`/`build.gradle` (Java).
- **tooling:** пакетный менеджер + запуск dev-сервера стека (из пака).
- **test_runner:** тест-раннер стека (pytest / jest / go test / JUnit — из пака).
- **build:** Docker-образ приложения с `entrypoint.sh`, который **сам применяет миграции** (`alembic upgrade head`/аналог) и затем стартует сервер — контейнер поднимается рабочим без ручных шагов.
- **deploy_model:** контейнер за reverse-proxy (Nginx) + HTTPS на VPS, либо PaaS. Для fullstack — общий корневой `docker-compose.yml` (db+api+web), поднимаемый одной командой; миграции — в entrypoint, не отдельным шагом деплоя.
- **mental_model_ref:** `request_flow_overview` — как HTTP-запрос проходит слои.

## Слои одной фичи (порядок шагов первой фичи)

`schema (DTO) → domain (сущность+правила) → repository_interface → repository_impl → migration → usecase → router → test`

| Слой | Ответственность | Чего НЕ делает |
|------|------------------|----------------|
| schema / DTO | контракт запроса/ответа, валидация входа | хранение, бизнес-логика |
| domain | сущности и бизнес-правила, не зависят ни от чего | импорт фреймворка/ORM |
| repository (interface) | абстракция доступа к данным | конкретный драйвер БД |
| repository (impl) | реализация на ORM/драйвере | бизнес-правила |
| migration | схема БД (создание/изменение таблиц) | логика приложения |
| usecase / service | оркестрация сценария, транзакции | знание про HTTP/SQL-диалект |
| router / handler | принять запрос, вызвать usecase, вернуть ответ, маппинг ошибок в HTTP-коды | прямые запросы к БД, бизнес-логика |
| test | проверка фичи (happy-path + краевые случаи) | — |

> Стеки без отдельного слоя миграций (или с code-first схемой) опускают шаг `migration` — это уточняет пак.

## Скелет глав (chapter_skeleton)

| # | Глава | Результат |
|---|-------|-----------|
| 0 | prerequisites *(если junior/чистая машина)* | окружение готово (язык, Docker, git, SSH) |
| 1 | starting_project | репозиторий, структура слоёв, первый эндпоинт (`/health`), Docker, первый тест |
| 2 | данные/безопасность *(если есть БД/auth)* | подключение БД, миграции, аутентификация |
| 3.. | features (по числу доменных сущностей) | рабочее API сущностей с тестами |
| N | delivery | CI, секреты, Dockerfile+entrypoint (авто-миграции), прод-сервер, деплой, README проекта, smoke-тест |

## Контракт (для fullstack/наличия фронта)

Если проект fullstack — реализация каждого эндпоинта **сверяется с `book/contracts/<domain>.yaml`**: пути, коды, формы запроса/ответа должны совпадать с контрактом (он — источник правды для фронта). Расхождение «код vs контракт» — дефект.

## Чек-лист «слой не протёк»

1. В роутере/handler нет прямых обращений к ORM/БД.
2. В usecase нет HTTP-примитивов (Request/Response/статус-коды) — бизнес-ошибки маппятся в HTTP в роутере.
3. Domain не импортирует фреймворк/ORM.
4. Репозиторий объявлен абстракцией; реализация — отдельно.
