# Чистая архитектура для FastAPI (router → usecase → repository)

> Базовый источник истины по слоям. Используется скилами `book-architect` (при планировании глав) и `book-author` (при написании шагов).
> Дата актуальности: 2026-06.

## 1. Слои и зависимости

Зависимости направлены **внутрь** (правило зависимостей): внешние слои знают о внутренних, не наоборот.

```
HTTP  →  [ router ]  →  [ usecase / service ]  →  [ repository (interface) ]  →  [ DB ]
                              │
                              └─ оперирует [ domain (entities) ] и [ schemas (DTO) ]
```

| Слой | Каталог | Ответственность | Чего НЕ делает |
|------|---------|------------------|----------------|
| Router (presentation) | `app/api/` | Принять HTTP-запрос, валидировать вход (Pydantic), вызвать usecase, вернуть ответ | Бизнес-логика, прямые запросы к БД |
| Usecase (application) | `app/usecases/` | Оркестрация бизнес-сценария, транзакции, вызовы репозиториев | Знание про HTTP, знание про SQL-диалект |
| Domain (entities) | `app/domain/` | Сущности и бизнес-правила, не зависящие ни от чего | Импорт FastAPI, SQLAlchemy |
| Repository (interface) | `app/repositories/` | Контракт доступа к данным (абстракция) | Конкретный драйвер БД в интерфейсе |
| Infrastructure | `app/infrastructure/` | Реализации репозиториев, ORM-модели, клиенты | Бизнес-правила |
| Schemas (DTO) | `app/schemas/` | Pydantic-модели запросов/ответов | Логика хранения |

## 2. Рекомендуемая структура каталогов

```
app/
├── main.py                 # создание FastAPI-приложения, подключение роутеров
├── core/                   # конфиг, настройки, безопасность, DI
│   ├── config.py
│   └── security.py
├── api/                    # роутеры (presentation)
│   └── v1/
│       ├── users.py
│       └── auth.py
├── usecases/               # бизнес-сценарии
│   └── user/
│       ├── create_user.py
│       └── authenticate_user.py
├── domain/                 # сущности и бизнес-правила
│   └── user.py
├── repositories/           # интерфейсы доступа к данным
│   └── user_repository.py
├── infrastructure/         # реализации: ORM-модели, репозитории, БД-сессии
│   ├── db.py
│   └── user_repository_impl.py
└── schemas/                # Pydantic DTO
    └── user.py
```

## 3. Принцип внедрения зависимостей (DI)

- Роутер зависит от usecase через `Depends(...)`.
- Usecase зависит от **интерфейса** репозитория, а не от реализации.
- Конкретная реализация репозитория собирается в `core` / на старте приложения.

Это позволяет тестировать usecase с фейковым репозиторием без БД.

## 4. Чек-лист «слой не протёк»

1. В роутере нет ни одного обращения к ORM/сессии БД напрямую.
2. В usecase нет `Request`, `Response`, `HTTPException` со статус-кодами presentation-уровня (бизнес-ошибки → доменные исключения, маппятся в HTTP в роутере).
3. Domain-сущности не импортируют FastAPI/SQLAlchemy.
4. Репозиторий объявлен как абстракция (Protocol/ABC), реализация — в `infrastructure`.

## 5. Как это отражается в книге

`book-architect` обязан разложить разработку каждой фичи на шаги в порядке слоёв:
`schema → domain → repository (interface) → repository (impl) → usecase → router → test`.
`book-author` в каждом шаге показывает код только своего слоя и объясняет его место в потоке зависимостей.
