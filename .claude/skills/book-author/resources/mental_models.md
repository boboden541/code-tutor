# База знаний: ментальные модели (вид сверху)

> Источник истины для шага `request_flow_overview.md` и для объяснений «как всё связано».
> Дата актуальности: 2026-06. Цель — дать новичку «карту» до погружения в код.

## 1. Поток запроса через слои

```
            HTTP-запрос (JSON)
                  │
                  ▼
        ┌───────────────────┐   app/api/ (presentation)
        │      ROUTER       │   принять запрос, валидировать (схема),
        │  @router.post(...)│   вызвать usecase, поймать доменную ошибку → HTTP-код
        └─────────┬─────────┘
                  │ вызывает
                  ▼
        ┌───────────────────┐   app/usecases/ (application)
        │      USECASE      │   бизнес-сценарий: проверки, оркестрация,
        │   .execute(...)   │   решение о commit; знает ТОЛЬКО интерфейс репозитория
        └─────────┬─────────┘
                  │ через интерфейс (Protocol)
                  ▼
        ┌───────────────────┐   app/repositories/ (контракт)  +
        │    REPOSITORY     │   app/infrastructure/ (реализация на SQLAlchemy)
        │ interface → impl  │   единственное место с SQL/ORM
        └─────────┬─────────┘
                  │
                  ▼
            ┌───────────┐
            │    БД     │   PostgreSQL
            └───────────┘

  Сбоку (не в потоке, а «материал»):
   • app/domain/   — сущности и бизнес-правила (ни от чего не зависят)
   • app/schemas/  — DTO запроса/ответа (Pydantic)
   • app/core/     — конфиг, безопасность, общие зависимости
```

**Ответ** идёт обратно тем же путём: репозиторий вернул домен → usecase вернул результат → роутер отдал его как JSON (Pydantic сериализует).

## 2. Правило зависимостей (главное)

Зависимости направлены **внутрь**: внешние слои знают о внутренних, не наоборот.

```
api  →  usecases  →  repositories(интерфейс)  ←  infrastructure(реализация)
                          ▲
                       domain (в центре, не зависит ни от чего)
```

Поэтому:
- роутер НЕ ходит в БД напрямую;
- usecase НЕ знает про HTTP и про конкретный SQL;
- domain НЕ импортирует FastAPI/SQLAlchemy.

## 3. Как собираются зависимости (DI)

```
запрос → FastAPI видит Depends(get_session) и Depends(get_<usecase>)
       → создаёт сессию БД
       → создаёт репозиторий(сессия)
       → создаёт usecase(репозиторий)
       → передаёт готовый usecase в функцию-обработчик
```

Читателю: «вы не создаёте объекты руками в роутере — их собирает FastAPI через `Depends`, по цепочке сверху вниз».

## 4. Как использовать в книге

`book-author` в шаге `request_flow_overview.md`:
1. Показать диаграмму потока (раздел 1) и правило зависимостей (раздел 2).
2. Объяснить, что дальше каждый слой пишется отдельным шагом — и указать порядок (`schema → domain → repository → usecase → router → test`).
3. Не давать кода — только модель; код будет в шагах фичи.
