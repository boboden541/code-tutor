## Использование

```
/tutor-design [project_brief.md] [requirements.md] [stack_decision.md]
```

**Параметры:** пути к Файлу №1, №2 и №3 (по умолчанию `project_brief.md`, `requirements.md`, `stack_decision.md`).

## Пример

```
/tutor-design
/tutor-design project_brief.md requirements.md stack_decision.md
```

**Ожидаемый результат:**
- **три системно разных** визуальных направления-прототипа (сильная типографика, продающий UX, реалистичный контент, адаптив) в Claude Design (DesignSync) или Artifact (fallback) — пользователь выбирает одно;
- `design_spec.md` (Файл №4) — суть+эстетика, три направления и выбранное, дизайн-система выбранного (токены со шрифтовой парой/компоненты с состояниями/триггеры доверия/тон), матрица «экран×эндпоинт», спецификация каждого экрана, способ стилизации, ссылки на прототипы.

> Шаг выполняется **только при наличии frontend-трека** (frontend/fullstack из Файла №3). Для backend-only — пропусти и сразу `/tutor-plan`.

## Инструкция для LLM

### Этап 1: Загрузка роли
1. Прочитай `.claude/skills/ui-designer/SKILL.md` — персона и принципы.
2. Загрузи `.claude/skills/ui-designer/resources/screen_derivation.md` — как выводить экраны и матрицу покрытия.
3. Загрузи `.claude/skills/ui-designer/resources/screen_spec_schema.md` — поля каждого экрана.
4. Загрузи `.claude/skills/ui-designer/resources/design_system_spec.md` — что фиксировать в дизайн-системе.
5. Загрузи `.claude/skills/ui-designer/resources/art_direction.md` — планка качества: три разных направления, сильная типографика, продающий UX, реалистичный контент, адаптив, запреты.
6. Загрузи `.claude/skills/ui-designer/resources/prototype_guide.md` — как собрать три прототипа и дать выбрать.
7. Подгрузи глобальный навык `frontend-design` для дисциплины эстетики (палитра/типографика/фирменный элемент).
8. Сверь термины с `.claude/skills/book-architect/resources/naming_conventions.md`.

### Этап 2: Чтение входов
Прочитай Файлы №1, №2, №3, API-контракт(ы) `book/contracts/*.yaml` и индекс Knowledge Pack фронт-трека (read-only). Если тип проекта backend-only — сообщи, что дизайн-шаг не нужен, и предложи `/tutor-plan`. Если нет Файла №3 — предложи `/tutor-stack`.

### Этап 3: Вывод экранов (матрица покрытия)
1. Назови суть продукта (предмет/аудитория/задача) — из неё выводится эстетика.
2. По `screen_derivation.md`: выпиши все операции контракта и ролевые флоу; разложи операции по экранам; построй матрицу «экран×эндпоинт».
3. Проверь: нет непотреблённых эндпоинтов, нет разрывов в ролевых флоу.

### Этап 4: Спецификация экранов
Для каждого экрана заполни поля `screen_spec_schema.md`: маршрут, role-gating, api_calls, формы+валидация (зеркало контракта), парсинг данных, состояния loading/empty/error и pending/success/failure, навигация, компоненты.

### Этап 5: Дизайн-система + три арт-направления
1. По `design_system_spec.md`: выведи из сути токены (палитра/типографика **со шрифтовой парой**/spacing/радиусы/тени), компоненты **с состояниями** (Button: hover/focus-visible/disabled/pending) и — если продукт продаёт доверие — **триггеры доверия** (напр. RatingStars/ReviewCard/TrustBadge), тон; зафиксируй способ стилизации (дефолт — токены + CSS Modules).
2. По `art_direction.md` спроектируй **три системно разных направления** (≥3 осей различия — типографика/композиция/эмоция, не только цвет): ярлык + одно-строчное позиционирование + шрифтовая пара + композиционный приём + характер CTA/hover для каждого.

### Этап 6: Три прототипа и выбор
По `prototype_guide.md` собери **три прототипа-направления** (реалистичный контент, продающие блоки, видимые состояния, адаптив) и дай **выбрать одно**:
- основная поверхность — **Claude Design (DesignSync)**: `list_projects` → (`create_project`) → `finalize_plan` → `write_files`; три направления как отдельные группы; дай ссылку;
- если DesignSync/ design-login недоступны — **fallback Artifact** (три self-contained HTML или один с переключателем); если и он недоступен — spec-only, честно отметь.
> Инструмент `DesignSync` может быть отложенным — найди его схему через ToolSearch (`DesignSync` / `design`) перед первым вызовом.
> Перед показом прогони каждый прототип по запретам `art_direction.md §F` (нет устаревших паттернов и слепых зон UX).
Дружелюбный чекпоинт: покажи 3 варианта, дефолт «выбрать рекомендованное и продолжить»; внеси правки пользователя.

### Этап 7: Файл №4 и чекпоинт
1. Сохрани `design_spec.md` по шаблону `.claude/skills/ui-designer/examples/ideal_design_spec.md` (суть+эстетика, **три направления и выбранное**, дизайн-система выбранного, матрица, экраны, способ стилизации, ссылки на прототипы/пометка fallback).
2. Выведи: «Файл №4 создан, направление выбрано. Следующий шаг: `/tutor-plan`».

> **Важно:** план фронт-книги нельзя строить без утверждённого `design_spec.md` — иначе книга даст белую страницу. Этот шаг — обязательный фундамент интерфейса между стеком и планом (для frontend/fullstack).
