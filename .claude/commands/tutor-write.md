## Использование

```
/tutor-write [путь к book/outline.md] [N] [--level junior|middle]
```

**Параметры:**
- путь к плану (по умолчанию `book/outline.md`);
- `N` — необязательно: сколько незавершённых шагов написать за этот вызов (по умолчанию — все `[ ]`);
- `--level` — необязательно: глубина подачи. По умолчанию берётся из профиля читателя Файла №2. `junior` → полные файлы, максимум разборов, «под капотом»; `middle` → допустимы diff и «по образцу со ссылкой».

## Пример

```
/tutor-write
/tutor-write book/outline.md 3
```

**Ожидаемый результат:** созданные `.md`-файлы шагов и обновлённый `outline.md` (`[ ]` → `[x]`).

## Инструкция для LLM

### Этап 1: Загрузка роли
1. Прочитай `.claude/skills/book-author/SKILL.md` — твоя персона и принципы.
2. Загрузи шаблон `.claude/skills/book-author/examples/ideal_section.md`.
3. Сверь формат и термины с `.claude/skills/book-architect/resources/naming_conventions.md`.

### Этап 2: Чтение плана и фундамента (возобновляемость)
1. Прочитай Файл №3 (`stack_decision.md`): тип проекта, стек(и), профиль(и), путь к **замороженному Knowledge Pack** (`book/_knowledge/`) и контракту (`book/contracts/`). Если пак не заморожен или Файла №3 нет — сообщи и предложи `/tutor-stack` (писать против незамороженного пака запрещено).
2. Прочитай `book/outline.md`. Распарси строки регэкспом `^- \[( |x)\] (\S+\.md) — (.+)$`.
3. Отбери шаги со статусом `[ ]`. Шаги `[x]` **не трогай** (идемпотентность).
4. Если все `[x]` — сообщи «Книга написана полностью» и предложи `/tutor-validate`.

### Этап 3: Посекционная запись (один шаг за итерацию)
Сначала определи `--level` (из параметра или профиля читателя Файла №2; по умолчанию `junior`). Для каждого выбранного `[ ]`-шага (не более `N`, если задано):
1. Определи трек шага (backend/frontend по части/папке) и стек этого трека.
2. Загрузи нужный файл **Knowledge Pack** `book/_knowledge/<tech>/` по теме шага + профиль из Файла №3; для API-шагов — `book/contracts/<domain>.yaml`.
3. Сгенерируй самодостаточный `.md` по шаблону `ideal_section.md`, СОБЛЮДАЯ принципы `1a`/`1b` book-author:
   - **полный, копируемый код** — без `...`, «фрагментов» и «по образцу без ссылки»;
   - после каждого нетривиального блока — раздел **«Разбор кода»** (все новые конструкции объяснены, термины — из `<pack>/glossary.md`);
   - где есть неявное поведение — блок **«Под капотом»**;
   - все имена в коде определены/собраны в этом же шаге; правка существующего файла — полный файл или diff с якорями;
   - конкретику (имена API, версии, команды) бери ТОЛЬКО из пака — не из общих знаний;
   - для API-шагов сверяйся с контрактом (пути/коды/формы/типы);
   - для `junior` — максимум разборов и полные файлы; для `middle` — допустимы diff и «по образцу со ссылкой на 1-ю фичу».
4. Запиши файл по точному пути из строки плана (с префиксом `<KK>_`, создай каталоги при необходимости).
5. Заголовок «Шаг N из M в секции» проставь по факту: `N` = префикс файла `<KK>`, `M` = число шагов в секции (по `outline.md`). Ссылки «Что дальше»/«Предусловия» — на файлы с их префиксами.
6. Обнови строку в `outline.md`: `[ ]` → `[x]`.

### Этап 4: Завершение
Выведи, сколько шагов написано и сколько осталось `[ ]`. Если остались — напомни, что повторный запуск `/tutor-write` допишет их. Если все готовы — предложи `/tutor-validate book/outline.md`.

> **Важно (идемпотентность):** уже отмеченные `[x]` файлы перезаписывать запрещено. Повторный запуск меняет только незавершённые шаги. Knowledge Pack — read-only (его собирает `/tutor-stack`).
