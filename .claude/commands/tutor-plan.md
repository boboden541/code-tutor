## Использование

```
/tutor-plan <project_brief.md> <requirements.md>
```

**Параметры:** пути к Файлу №1 и Файлу №2 (по умолчанию `project_brief.md` и `requirements.md`).

## Пример

```
/tutor-plan project_brief.md requirements.md
```

**Ожидаемый результат:** файл `book/outline.md` — план книги (главы → секции → шаги) в строгом машиночитаемом формате.

## Инструкция для LLM

### Этап 1: Загрузка роли
1. Прочитай `.claude/skills/book-architect/SKILL.md` — твоя персона и принципы.
2. Загрузи формат `.claude/skills/book-architect/examples/ideal_outline.md`.
3. Загрузи правила `.claude/skills/book-architect/resources/delivery_pipeline.md`.
4. Сверь термины и формат с `.claude/skills/book-architect/resources/naming_conventions.md`.

### Этап 2: Чтение входов
Прочитай Файл №1 и Файл №2. Если какого-то нет — сообщи и предложи предыдущий шаг.

### Этап 3: Построение плана
1. **Глава 0** (`chapter_0_prerequisites`): если профиль читателя в Файле №2 — junior/чистая машина, добавь её первой (установка Python/Docker/git/SSH, см. `prerequisites.md`). Для middle — пропусти.
2. Разбей остальную разработку на главы (порядок: старт → первое API/Docker → домен и API сущностей → безопасность → тесты → CI/GitLab → деплой на прод).
3. Первая доменная глава открывается секцией `section_0_request_flow_overview` с шагом `request_flow_overview.md` (см. `mental_models.md`).
4. **Первая доменная фича — ОДИН СЛОЙ = ОДИН ШАГ** (`<f>_schema → _domain → _repository_interface → _repository_impl → _migration → _usecase → _router → _test`); запрет сворачивать в `*_feature.md`. Вторая фича — слои сгруппированы; третья+ — «по образцу со ссылкой» (см. `delivery_pipeline.md`, `SKILL.md ## 4a`).
5. Каждая секция-фича обрамляется `start_feature_branch` (первый шаг) и `commit_push_open_mr` (последний); глава 1 — памятка `how_we_work_git_flow`.
6. **Каждая глава завершается** шагом `recap_and_state.md` (что поняли + полный `requirements.txt` + дерево).
7. Каждый файл-шаг именуй с **2-значным префиксом порядка** `<KK>_<name>.md` (`01_`, `02_`, …), сквозным с `01` внутри секции — порядок виден в папке (см. `naming_conventions.md`).
8. Каждый шаг — отдельная строка строгого формата `- [ ] <путь>.md — <назначение>` (регэксп `^- \[( |x)\] (\S+\.md) — (.+)$`); строки секции идут по возрастанию префикса.
9. Каждый пункт требований из Файла №2 должен быть покрыт хотя бы одним шагом.

### Этап 4: Дружелюбный чекпоинт
1. Выведи оглавление в чат (главы → секции → шаги с одной строкой назначения).
2. Сообщи: «Можно поправить план или просто продолжить (дефолт)». Не блокируй пользователя.

### Этап 5: Сохранение
1. Сохрани план в `book/outline.md` (все шаги со статусом `[ ]`).
2. Выведи: «План книги создан: `book/outline.md`. Следующий шаг: `/tutor-write` (генерация секций). Запускайте его столько раз, сколько нужно — он дописывает недостающие шаги».
