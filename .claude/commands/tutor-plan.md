## Использование

```
/tutor-plan [project_brief.md] [requirements.md] [stack_decision.md]
```

**Параметры:** пути к Файлу №1, Файлу №2 и Файлу №3 (по умолчанию `project_brief.md`, `requirements.md`, `stack_decision.md`).

## Пример

```
/tutor-plan
/tutor-plan project_brief.md requirements.md stack_decision.md
```

**Ожидаемый результат:** файл `book/outline.md` — план книги (части → главы → секции → шаги) в строгом машиночитаемом формате.

## Инструкция для LLM

### Этап 1: Загрузка роли
1. Прочитай `.claude/skills/book-architect/SKILL.md` — твоя персона и принципы.
2. Загрузи формат `.claude/skills/book-architect/examples/ideal_outline.md`.
3. Загрузи правила `.claude/skills/book-architect/resources/delivery_pipeline.md`.
4. Загрузи `.claude/skills/book-architect/resources/profiles/_profile_schema.md` и конкретные профили, указанные в Файле №3.
5. Сверь термины и формат с `.claude/skills/book-architect/resources/naming_conventions.md`.

### Этап 2: Чтение входов
Прочитай Файлы №1, №2, №3. Если Файла №3 нет — сообщи и предложи запустить `/tutor-stack` (без выбранного стека, профиля и замороженного пака план строить нельзя). Из Файла №3 выпиши: **тип проекта**, стек(и), резолвленный профиль(и), путь к Knowledge Pack и контракту.

### Этап 3: Построение плана
1. Загрузи `chapter_skeleton`, `layers`, `state_snapshot_file`, `mental_model_ref` из профиля(ей) Файла №3 — **структуру бери из профиля, не из хардкода FastAPI**.
2. **Глава 0** (`chapter_0_prerequisites`): если профиль читателя junior/чистая машина — добавь первой (установка инструментов выбранного стека).
3. Разбей разработку на главы по `chapter_skeleton` (порядок: старт/scaffold → фундамент → фичи → тесты → CI → деплой).
4. Первая доменная глава трека открывается обзором `mental_model_ref` (бэк → `request_flow_overview`; фронт → `data_flow_overview`).
5. **Первая фича трека — ОДИН СЛОЙ = ОДИН ШАГ** (порядок = `layers` профиля); запрет сворачивать в `*_feature.md`. Вторая фича — слои сгруппированы; третья+ — «по образцу со ссылкой».
6. **Fullstack:** введи части `part_1_backend/` и `part_2_frontend/`; после общего фундамента — контракт-first срезы по фичам (`define_api_contract` → бэк-реализация → фронт-потребление → `wire_end_to_end`). Контракт — в `book/contracts/`.
7. Каждая секция-фича обрамляется `start_feature_branch` / `commit_push_open_mr`; глава 1 — памятка `how_we_work_git_flow`.
8. **Каждая глава завершается** `recap_and_state.md` (что поняли + полный `state_snapshot_file` профиля + дерево).
9. Каждый файл-шаг — с **2-значным префиксом порядка** `<KK>_<name>.md`, сквозным с `01` внутри секции.
10. Каждый шаг — строка строгого формата `- [ ] <путь>.md — <назначение>` (регэксп `^- \[( |x)\] (\S+\.md) — (.+)$`); заголовки частей `# part_<P>_<track>`, глав `## chapter_<N>_<name>`.
11. Каждый пункт Файла №2 покрыт ≥1 шагом; каждый эндпоинт IR — контрактом + реализацией (+ потреблением для fullstack).

### Этап 4: Дружелюбный чекпоинт
1. Выведи оглавление в чат (части → главы → секции → шаги с одной строкой назначения).
2. Сообщи: «Можно поправить план или просто продолжить (дефолт)». Не блокируй пользователя.

### Этап 5: Сохранение
1. Сохрани план в `book/outline.md` (все шаги `[ ]`).
2. Выведи: «План книги создан: `book/outline.md`. Следующий шаг: `/tutor-write`. Запускайте его столько раз, сколько нужно — он дописывает недостающие шаги».
