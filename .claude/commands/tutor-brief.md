## Использование

```
/tutor-brief <описание проекта>
```

**Параметр:** `<описание>` — свободный текст: какой проект вы хотите (backend/frontend/fullstack). Стек указывать **не обязательно** — если не укажете, его предложит шаг `/tutor-stack`.

## Примеры

```
/tutor-brief Хочу fullstack-приложение «список задач»: пользователи, проекты, задачи. Бэкенд + веб-интерфейс. Стек на ваше усмотрение.
```

```
/tutor-brief Сервис заметок с тегами. Стек: FastAPI, PostgreSQL, React. Остальное предложите сами.
```

**Ожидаемый результат:** файл `project_brief.md` (Файл №1) в корне проекта.

## Инструкция для LLM

### Этап 1: Загрузка роли
1. Прочитай `.claude/skills/project-brief/SKILL.md` — твоя персона и принципы.
2. Загрузи шаблон `.claude/skills/project-brief/examples/ideal_brief.md`.
3. Сверь термины с `.claude/skills/book-architect/resources/naming_conventions.md`.

### Этап 2: Деконструкция ввода
Из аргумента `<описание>` извлеки: цель продукта, **тип проекта** (backend/frontend/fullstack), стек (если задан), ключевые сущности, сценарии, границы MVP, профиль читателя.

### Этап 3: Формирование Файла №1
1. Заполни `project_brief.md` строго по шаблону `ideal_brief.md`.
2. **Тип проекта** определи обязательно (backend/frontend/fullstack) — он управляет структурой книги.
3. **Стек:** если задан — переноси дословно, не подменяй дефолтным; если не задан — поставь `[предложит /tutor-stack]`, **не выдумывай**.
4. Прочие поля, не выводимые из ввода, помечай `[уточнить]` — НЕ выдумывай.

### Этап 4: Сохранение
1. Если `project_brief.md` уже существует — спроси, перезаписать ли.
2. Сохрани `project_brief.md` в корне проекта.
3. Выведи: «Файл №1 создан: `project_brief.md`. Проверьте тип проекта, стек (если задан) и поля `[уточнить]`/`[предложит /tutor-stack]`. Следующий шаг: `/tutor-requirements project_brief.md`».
